# PEBL Study Format Specification

## Version

**Specification Version:** 1.0
**Date:** 2026-01-09
**Status:** Draft

## Overview

This document specifies the file format and structure for PEBL studies in the native launcher. This format is designed to be:

1. **Self-contained** - Everything needed to run a study in one directory
2. **Portable** - Can be zipped and shared or uploaded
3. **Bidirectional** - Same format for native and online platform
4. **Human-readable** - JSON files can be edited manually if needed
5. **Version-controlled** - Works with git and other VCS

## Directory Structure

```
study-name/
├── study-info.json          # Required: Study metadata
├── chains/                  # Optional: Test chains
│   ├── chain-name.json
│   └── another-chain.json
└── tests/                   # Required: Test directories
    ├── test-name-1/
    │   ├── test.pbl
    │   ├── params/
    │   │   ├── test.pbl.schema.json
    │   │   └── variant.par.json
    │   ├── translations/
    │   │   ├── test.pbl-en.json
    │   │   └── test.pbl-es.json
    │   └── data/            # Created at runtime, excluded from snapshots
    │       └── subject-id/
    └── test-name-2/
        └── ...
```

## File Naming Conventions

**CRITICAL:** Never use spaces in filenames or directory names.

### Directory Names
- Study directory: `study-name` (lowercase, hyphens)
- Test directory: follows battery conventions (usually `TestName` or `testname`)
- Chain directory: `chains` (fixed name)

### File Names
- Study metadata: `study-info.json` (fixed name)
- Chain files: `chain-name.json` (lowercase, hyphens, .json extension)
- Parameter files: `test-variant.par.json` (hyphens, .par.json extension)
- Schema files: `test.pbl.schema.json` (fixed suffix)
- Translation files: `test.pbl-lang.json` (two-letter language code)

### Subject IDs
- Format: alphanumeric with hyphens/underscores only
- Valid: `101`, `subject-101`, `pilot_01`
- Invalid: `pilot 01` (space), `subject 101` (space)

## study-info.json Format

### Required Fields

```json
{
  "study_name": "string",    // Display name (can contain spaces)
  "version": 1,              // Integer version number
  "tests": []                // Array of test objects
}
```

### Complete Structure

```json
{
  "study_name": "Study Display Name",
  "description": "Optional study description",
  "version": 1,
  "created_date": "2026-01-09T12:00:00Z",
  "modified_date": "2026-01-09T15:30:00Z",
  "author": "Lab Name or Researcher",
  "study_token": "",
  "tests": [
    {
      "test_name": "TestName",
      "test_path": "tests/TestName",
      "included": true,
      "parameter_variants": {
        "default": {
          "description": "Default parameters",
          "file": null
        },
        "variant-name": {
          "description": "Variant description",
          "file": "params/TestName-variant.par.json"
        }
      }
    }
  ]
}
```

### Field Descriptions

#### study_name
- **Type:** String
- **Required:** Yes
- **Description:** Display name shown in UI
- **Constraints:** Non-empty
- **Notes:** Can contain spaces and special characters

#### description
- **Type:** String
- **Required:** No
- **Description:** Study purpose or notes
- **Default:** Empty string

#### version
- **Type:** Integer
- **Required:** Yes
- **Description:** Study version number
- **Constraints:** >= 1
- **Notes:** Increment when making significant changes

#### created_date
- **Type:** String (ISO 8601)
- **Required:** No (but recommended)
- **Format:** `YYYY-MM-DDTHH:MM:SSZ`
- **Example:** `"2026-01-09T12:00:00Z"`

#### modified_date
- **Type:** String (ISO 8601)
- **Required:** No (but recommended)
- **Format:** `YYYY-MM-DDTHH:MM:SSZ`
- **Notes:** Update whenever study is changed

#### author
- **Type:** String
- **Required:** No
- **Description:** Creator or lab name

#### study_token
- **Type:** String
- **Required:** No
- **Description:** Platform-generated access token
- **Default:** Empty string for native studies
- **Notes:** Generated by online platform on upload

#### tests
- **Type:** Array of test objects
- **Required:** Yes
- **Constraints:** Can be empty but must exist
- **Description:** List of tests included in study

### Test Object Structure

Each test in the `tests` array has this structure:

```json
{
  "test_name": "TestName",
  "test_path": "tests/TestName",
  "included": true,
  "parameter_variants": {
    "default": {
      "description": "Default parameters",
      "file": null
    }
  }
}
```

#### test_name
- **Type:** String
- **Required:** Yes
- **Description:** Name of test (matches directory name)
- **Example:** `"BART"`, `"corsi"`, `"stroop"`

#### test_path
- **Type:** String
- **Required:** Yes
- **Description:** Relative path to test directory
- **Format:** `"tests/TestName"`
- **Notes:** Path is relative to study root

#### included
- **Type:** Boolean
- **Required:** Yes
- **Description:** Whether test is active
- **Default:** `true`
- **Notes:** Allows temporarily disabling tests

#### parameter_variants
- **Type:** Object (map of variant names to variant objects)
- **Required:** Yes
- **Constraints:** Must have at least "default"
- **Description:** Available parameter configurations

### Parameter Variant Object

```json
{
  "description": "Human-readable description",
  "file": "path/to/variant.par.json" or null
}
```

#### Variant Keys
- **Format:** lowercase-with-hyphens
- **Special:** "default" is required
- **Examples:** `"default"`, `"large"`, `"short-version"`

#### description
- **Type:** String
- **Required:** Yes
- **Description:** Human-readable variant description

#### file
- **Type:** String or null
- **Required:** Yes
- **Description:** Path to .par.json file
- **Null for default:** If null, test uses built-in defaults
- **Path format:** Relative to test directory

## Validation Rules

### study-info.json Validation

1. **Must be valid JSON** - Parseable by standard JSON parser
2. **Required fields present:**
   - `study_name` (non-empty string)
   - `version` (integer >= 1)
   - `tests` (array)
3. **Date fields if present:**
   - Must be valid ISO 8601 format
   - `modified_date` >= `created_date`
4. **Test objects:**
   - Each test has required fields
   - `test_path` directory exists
   - Parameter variants include "default"
5. **No duplicate test names**

### Recommendations (Warnings)

1. Include `description` field
2. Include `author` field
3. Include `created_date` and `modified_date`
4. At least one test in `tests` array
5. All `included` tests have matching directories

## Parameter Variant Files

Parameter variant files (`.par.json`) override default test parameters.

### Format

```json
{
  "parameter_name": value,
  "another_parameter": value
}
```

### Example

```json
{
  "trials": 30,
  "timeout": 3000,
  "feedback": true,
  "difficulty": "hard"
}
```

### Rules

1. **Valid JSON object** - Top-level must be object
2. **Parameter names** - Must match test's schema
3. **Type matching** - Values must match expected types
4. **Only overrides** - Don't need to include all parameters
5. **No nested objects** - Flat parameter structure

## Schema Files

Schema files (`.pbl.schema.json`) define available parameters for a test.

### Format

Pipe-delimited text file, one parameter per line:

```
parameter_name|default_value|description
```

### Example

```
trials|20|Number of trials to run
timeout|5000|Response timeout in milliseconds
feedback|1|Show feedback after each trial (0=no, 1=yes)
difficulty|medium|Task difficulty level (easy, medium, hard)
```

### Rules

1. **Three fields required:** name, default, description
2. **Delimiter:** Pipe character (`|`)
3. **No header row**
4. **Comments:** Lines starting with `#` ignored
5. **Empty lines:** Ignored
6. **Type inference:** From default value
   - Integer: `20`, `100`
   - Float: `2.5`, `100.0`
   - Boolean: `0`, `1`
   - String: `"text"` or non-numeric

## Directory Requirements

### Required Directories

1. **Study root** - Top-level directory
2. **tests/** - Contains test directories

### Optional Directories

1. **chains/** - Chain definitions
2. **tests/TestName/params/** - Parameter files
3. **tests/TestName/translations/** - Translation files
4. **tests/TestName/data/** - Runtime data (not in snapshots)

### Prohibited in Snapshots

1. **tests/*/data/** - Participant data must be excluded
2. **Temporary files** - No .tmp, .bak, etc.
3. **Version control** - No .git, .svn directories
4. **OS files** - No .DS_Store, Thumbs.db, desktop.ini

## Snapshot vs Working Study

### Working Study
- Located in `pebl-exp.2.3/my_studies/`
- Contains `tests/*/data/` directories
- Participant data accumulates
- Modified during use

### Snapshot
- Located in `pebl-exp.2.3/snapshots/`
- No `data/` directories
- Clean and shareable
- Ready for upload/distribution
- Created from working study

### Snapshot Creation Rules

1. **Copy structure** - Preserve directory layout
2. **Exclude data/** - Remove all `tests/*/data/` directories
3. **Include tests** - Full test directories (code, params, translations)
4. **Include chains** - All chain JSON files
5. **Clean files** - No temporary or system files

## Version Management

### Version Number

- **Integer only** - No decimals (1, 2, 3, not 1.0, 1.1)
- **Increment on:**
  - Adding/removing tests
  - Changing chains
  - Significant parameter changes
  - Breaking changes
- **Don't increment for:**
  - Minor text edits
  - Typo fixes
  - Adding comments

### Version History

Studies don't maintain version history internally. Use:
- Git for version control
- Snapshots for major versions
- File naming for snapshot versions: `study_v1_date`, `study_v2_date`

## File Size Considerations

### Recommended Limits

- **Study-info.json:** < 1 MB
- **Chain file:** < 100 KB
- **Total study (without data):** < 500 MB
- **Individual test:** < 50 MB

### Large Files

For tests with large media files:
- Keep media in test directory
- Media included in snapshots
- Consider compression for images
- Use appropriate formats (PNG, MP3, etc.)

## Character Encoding

- **All JSON files:** UTF-8 encoding
- **No BOM:** Byte Order Mark not required
- **Line endings:** Any (LF, CRLF) accepted
- **Special characters:** Properly escaped in JSON

## JSON Formatting

### Recommended Style

- **Indentation:** 2 spaces (not tabs)
- **Trailing commas:** Not allowed (JSON spec)
- **Quotes:** Double quotes for strings
- **Booleans:** `true`/`false` (lowercase)
- **Null:** `null` (lowercase)

### Example (Formatted)

```json
{
  "study_name": "Example Study",
  "version": 1,
  "tests": [
    {
      "test_name": "BART",
      "test_path": "tests/BART",
      "included": true,
      "parameter_variants": {
        "default": {
          "description": "Standard parameters",
          "file": null
        }
      }
    }
  ]
}
```

## Error Handling

### Invalid JSON

If JSON parsing fails:
1. Show clear error message
2. Indicate which file failed
3. Don't crash - gracefully handle
4. Offer to open file in editor

### Missing Required Fields

If required fields missing:
1. List missing fields
2. Explain requirement
3. Offer to create with defaults
4. Don't allow save until fixed

### Invalid Paths

If test paths don't exist:
1. List missing paths
2. Offer to remove from study
3. Allow manual path correction
4. Verify before running

## Compatibility

### Forward Compatibility

Newer launchers should:
- Ignore unknown fields
- Use defaults for missing optional fields
- Warn about unrecognized values

### Backward Compatibility

When adding features:
- Make new fields optional
- Provide sensible defaults
- Don't break old studies
- Version specification clearly
